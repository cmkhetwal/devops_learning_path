# Lesson 04: Elastic Compute Cloud (EC2)

## Why This Matters in DevOps

EC2 is the workhorse of AWS. Despite the rise of containers and serverless, EC2 remains
the backbone of most production workloads. Understanding EC2 means understanding virtual
machine lifecycle management, capacity planning, cost optimization, and automated scaling
--- all core DevOps competencies.

As a DevOps engineer, you will provision EC2 instances through Terraform, bootstrap them
with user data scripts, monitor them with CloudWatch, and scale them with Auto Scaling
Groups. You will choose instance types based on workload characteristics, select pricing
models based on usage patterns, and design architectures that survive instance failures.

EC2 is also the foundation for understanding other AWS compute services. ECS runs
containers on EC2 (or Fargate). EKS runs Kubernetes nodes on EC2. Even Lambda runs
on micro-VMs under the hood. Master EC2, and every other compute service becomes easier.

## Core Concepts

### Instance Types

AWS instance types follow a naming convention: **Family + Generation + Size**

```
  m   5   .   xlarge
  |   |       |
  |   |       +-- Size: nano, micro, small, medium, large, xlarge, 2xlarge...
  |   +---------- Generation: higher = newer and better price/performance
  +-------------- Family: workload optimization category
```

| Family | Optimized For | Use Cases | Examples |
|--------|--------------|-----------|---------|
| **t3/t3a** | Burstable general purpose | Dev/test, small web apps, microservices | t3.micro, t3.medium |
| **m5/m6i** | General purpose (balanced) | App servers, backend, mid-size DBs | m5.large, m6i.xlarge |
| **c5/c6i** | Compute optimized | Batch processing, gaming, ML inference, HPC | c5.2xlarge |
| **r5/r6i** | Memory optimized | In-memory caches, real-time analytics, large DBs | r5.4xlarge |
| **i3/i4i** | Storage optimized | Data warehousing, distributed file systems | i3.large |
| **p4/g5** | GPU accelerated | ML training, video encoding, 3D rendering | p4d.24xlarge |

**T3 Burst Credits**: T3 instances earn CPU credits when idle and spend them when
busy. Great for workloads with variable CPU usage. If you consistently exceed your
baseline, switch to m5 (unlimited mode or fixed performance).

### AMIs (Amazon Machine Images)

```
AMI LIFECYCLE:
                                    +---> Instance 1
  Base AMI --> Customize --> Create AMI --> Instance 2
  (Amazon     (install       (Golden       +---> Instance 3
   Linux 2)    your app)      Image)

TYPES:
  - AWS-provided:    Amazon Linux 2, Ubuntu, Windows Server
  - Marketplace:     Pre-configured by vendors (with software)
  - Community:       Shared by other users (use with caution)
  - Custom (yours):  Your golden images with your app pre-installed
```

### Instance Lifecycle

```
  pending --> running --> stopping --> stopped --> terminated
                |                        |
                +-- rebooting -----------+
                |
                +-- shutting-down --> terminated

  IMPORTANT:
  - stopped: No compute charges (EBS still charged)
  - terminated: Everything deleted (unless EBS set to persist)
  - You cannot go from terminated back to running
```

### Instance Store vs EBS

```
INSTANCE STORE (Ephemeral):        EBS (Persistent):
+-------------------------+       +-------------------------+
| Physically attached     |       | Network-attached        |
| to host machine         |       | block storage           |
| FASTEST I/O             |       | Persists independently  |
| DATA LOST on stop/term  |       | Can snapshot and backup |
| Cannot detach/reattach  |       | Can resize on the fly   |
| Good for: temp data,    |       | Good for: OS, databases |
| caches, scratch space   |       | app data, boot volumes  |
+-------------------------+       +-------------------------+
```

**EBS Volume Types:**

| Type | IOPS | Throughput | Use Case |
|------|------|-----------|----------|
| gp3 (General SSD) | 3,000-16,000 | 125-1,000 MB/s | Boot volumes, dev/test, most workloads |
| io2 (Provisioned SSD) | Up to 64,000 | 1,000 MB/s | Databases needing consistent IOPS |
| st1 (Throughput HDD) | 500 | 500 MB/s | Big data, log processing, streaming |
| sc1 (Cold HDD) | 250 | 250 MB/s | Infrequent access, lowest cost |

### Pricing Models

```
PRICING COMPARISON (approximate for m5.large, us-east-1):

  On-Demand:     $0.096/hr    ($70/month)    -- Pay as you go
  Reserved 1yr:  $0.060/hr    ($44/month)    -- ~37% savings
  Reserved 3yr:  $0.040/hr    ($29/month)    -- ~58% savings
  Spot:          $0.029/hr    ($21/month)    -- ~70% savings (can be interrupted)
  Savings Plans: $0.055/hr    ($40/month)    -- Flexible commitment discount
```

| Model | Best For | Risk |
|-------|---------|------|
| **On-Demand** | Unpredictable workloads, short-term, dev/test | No risk, highest cost |
| **Reserved** | Steady-state, predictable workloads (databases, core apps) | Committed for 1-3 years |
| **Spot** | Fault-tolerant, flexible workloads (batch, CI/CD runners, data analysis) | Can be terminated with 2-min warning |
| **Savings Plans** | Predictable compute spend across instance families | Committed hourly spend |

### Placement Groups

| Strategy | Description | Use Case |
|---------|-------------|----------|
| **Cluster** | Instances packed on same rack | Low latency, HPC, tightly coupled |
| **Spread** | Instances on different hardware | High availability, critical instances |
| **Partition** | Groups on separate racks | Large distributed systems (Hadoop, Kafka) |

### User Data (Bootstrap Scripts)

User data runs once at instance launch. Use it to automate instance configuration:

```bash
#!/bin/bash
# This script runs as root at first boot
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd
echo "<h1>Hello from $(hostname -f)</h1>" > /var/www/html/index.html
```

## Step-by-Step Practical

### Deploy a Web Application on EC2

```bash
# Step 1: Create a key pair for SSH access
aws ec2 create-key-pair \
    --key-name devops-key \
    --query 'KeyMaterial' \
    --output text > devops-key.pem
chmod 400 devops-key.pem

# Step 2: Find the latest Amazon Linux 2023 AMI
AMI_ID=$(aws ec2 describe-images \
    --owners amazon \
    --filters "Name=name,Values=al2023-ami-2023.*-x86_64" \
              "Name=state,Values=available" \
    --query 'Images | sort_by(@, &CreationDate) | [-1].ImageId' \
    --output text)
echo "AMI: $AMI_ID"

# Step 3: Launch an EC2 instance with user data
cat > userdata.sh << 'USERDATA'
#!/bin/bash
dnf update -y
dnf install -y httpd
systemctl start httpd
systemctl enable httpd

cat > /var/www/html/index.html << 'HTML'
<!DOCTYPE html>
<html>
<head><title>DevOps Web App</title></head>
<body>
<h1>Hello from EC2!</h1>
<p>Instance ID: $(curl -s http://169.254.169.254/latest/meta-data/instance-id)</p>
<p>AZ: $(curl -s http://169.254.169.254/latest/meta-data/placement/availability-zone)</p>
</body>
</html>
HTML
USERDATA

INSTANCE_ID=$(aws ec2 run-instances \
    --image-id $AMI_ID \
    --instance-type t3.micro \
    --key-name devops-key \
    --security-group-ids $WEB_SG \
    --subnet-id $PUB_SUB_1 \
    --iam-instance-profile Name=EC2-S3-ReadOnly-Profile \
    --user-data file://userdata.sh \
    --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=web-server-1}]' \
    --query 'Instances[0].InstanceId' --output text)

# Step 4: Wait for instance to be running
aws ec2 wait instance-running --instance-ids $INSTANCE_ID

# Step 5: Get the public IP
PUBLIC_IP=$(aws ec2 describe-instances \
    --instance-ids $INSTANCE_ID \
    --query 'Reservations[0].Instances[0].PublicIpAddress' --output text)
echo "Access your app at: http://$PUBLIC_IP"

# Step 6: SSH into the instance
ssh -i devops-key.pem ec2-user@$PUBLIC_IP
```

### Create a Launch Template and Auto Scaling Group

```bash
# Create a Launch Template
TEMPLATE_ID=$(aws ec2 create-launch-template \
    --launch-template-name web-app-template \
    --version-description "v1" \
    --launch-template-data '{
        "ImageId": "'$AMI_ID'",
        "InstanceType": "t3.micro",
        "KeyName": "devops-key",
        "SecurityGroupIds": ["'$WEB_SG'"],
        "UserData": "'$(base64 -w0 userdata.sh)'",
        "IamInstanceProfile": {"Name": "EC2-S3-ReadOnly-Profile"},
        "TagSpecifications": [{
            "ResourceType": "instance",
            "Tags": [{"Key": "Name", "Value": "web-asg-instance"}]
        }]
    }' --query 'LaunchTemplate.LaunchTemplateId' --output text)

# Create the Auto Scaling Group
aws autoscaling create-auto-scaling-group \
    --auto-scaling-group-name web-app-asg \
    --launch-template LaunchTemplateId=$TEMPLATE_ID,Version='$Latest' \
    --min-size 2 \
    --max-size 4 \
    --desired-capacity 2 \
    --vpc-zone-identifier "$PUB_SUB_1,$PUB_SUB_2" \
    --health-check-type ELB \
    --health-check-grace-period 300 \
    --tags Key=Environment,Value=production,PropagateAtLaunch=true

# Create a target tracking scaling policy
aws autoscaling put-scaling-policy \
    --auto-scaling-group-name web-app-asg \
    --policy-name cpu-target-tracking \
    --policy-type TargetTrackingScaling \
    --target-tracking-configuration '{
        "PredefinedMetricSpecification": {
            "PredefinedMetricType": "ASGAverageCPUUtilization"
        },
        "TargetValue": 70.0
    }'

# Verify the ASG
aws autoscaling describe-auto-scaling-groups \
    --auto-scaling-group-names web-app-asg \
    --query 'AutoScalingGroups[0].[MinSize,MaxSize,DesiredCapacity,Instances[*].InstanceId]' \
    --output table
```

### Create an EBS Snapshot and AMI

```bash
# Create a snapshot of the root volume
VOLUME_ID=$(aws ec2 describe-instances \
    --instance-ids $INSTANCE_ID \
    --query 'Reservations[0].Instances[0].BlockDeviceMappings[0].Ebs.VolumeId' \
    --output text)

aws ec2 create-snapshot \
    --volume-id $VOLUME_ID \
    --description "Web server backup" \
    --tag-specifications 'ResourceType=snapshot,Tags=[{Key=Name,Value=web-server-backup}]'

# Create a custom AMI (golden image)
aws ec2 create-image \
    --instance-id $INSTANCE_ID \
    --name "web-app-golden-$(date +%Y%m%d)" \
    --description "Web app with Apache pre-configured" \
    --no-reboot
```

## Knowledge Check

### Question 1
What happens to data on an instance store volume when an EC2 instance is stopped?

**Answer:** The data is lost. Instance store volumes are ephemeral --- they are
physically attached to the host machine, and when an instance stops, it may restart
on a different host. Data is only preserved across reboots, not stops or terminations.
Use EBS volumes for data that must persist.

### Question 2
When would you choose a Spot Instance over On-Demand?

**Answer:** Choose Spot Instances for workloads that are fault-tolerant and flexible
in when they run: batch processing, CI/CD build runners, data analysis, image
rendering, and test environments. Spot Instances can be interrupted with a 2-minute
warning, so they are unsuitable for databases, stateful applications, or anything
requiring continuous availability.

### Question 3
What is the difference between a Launch Configuration and a Launch Template?

**Answer:** Launch Templates are the newer, recommended replacement for Launch
Configurations. Templates support versioning, can be used outside of ASGs (to
launch standalone instances), support multiple instance types in a single template,
and support advanced features like Spot and On-Demand mixed instances. Launch
Configurations are legacy, immutable (you must recreate them to change), and
cannot be versioned.

### Question 4
How does an Auto Scaling Group health check work?

**Answer:** ASGs support two health check types: EC2 (default) checks if the instance
is running at the hypervisor level; ELB checks if the instance passes the load
balancer's health check (e.g., HTTP 200 on /health). If an instance fails its health
check after the grace period, the ASG terminates it and launches a replacement. The
grace period (default 300s) prevents the ASG from terminating instances that are
still initializing.

### Question 5
Explain the t3 CPU credit system.

**Answer:** T3 instances have a baseline CPU performance (e.g., t3.micro = 10%).
When CPU usage is below baseline, the instance earns credits. When usage exceeds
baseline, it spends credits. With standard mode, the instance is throttled to
baseline when credits run out. With unlimited mode (default), the instance can
burst beyond baseline but you are charged for surplus credits. T3 is ideal for
workloads with occasional spikes but low average utilization.
