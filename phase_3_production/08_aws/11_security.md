# Lesson 11: AWS Security Services

## Why This Matters in DevOps

Security is not a feature you add at the end --- it is a practice woven into every
layer of your infrastructure. In DevOps, the term "DevSecOps" emerged because
security must shift left: integrated into CI/CD pipelines, infrastructure code,
and operational procedures from day one.

AWS provides dozens of security services, each addressing a specific threat vector.
As a DevOps engineer, you must know which tool to use for which problem: KMS for
encryption keys, Secrets Manager for credentials, GuardDuty for threat detection,
WAF for web application protection, and Shield for DDoS mitigation.

The SA Associate exam heavily tests security concepts. Every architecture question
has a security dimension: How do you encrypt data at rest? How do you protect a
public-facing API? How do you manage database passwords without hardcoding them?
How do you detect compromised credentials? This lesson gives you the knowledge to
answer those questions and implement the solutions.

## Core Concepts

### Defense in Depth

```
DEFENSE IN DEPTH: Multiple layers of security

  +----------------------------------------------------------+
  | Layer 1: EDGE SECURITY                                    |
  | CloudFront + WAF + Shield (DDoS protection)               |
  +----------------------------------------------------------+
  | Layer 2: NETWORK SECURITY                                 |
  | VPC, Security Groups, NACLs, Private Subnets              |
  +----------------------------------------------------------+
  | Layer 3: IDENTITY & ACCESS                                |
  | IAM Policies, Roles, MFA, SCP                             |
  +----------------------------------------------------------+
  | Layer 4: APPLICATION SECURITY                             |
  | Input validation, WAF rules, API throttling               |
  +----------------------------------------------------------+
  | Layer 5: DATA SECURITY                                    |
  | KMS encryption, SSL/TLS, Secrets Manager                  |
  +----------------------------------------------------------+
  | Layer 6: DETECTION & RESPONSE                             |
  | GuardDuty, Security Hub, CloudTrail, Config               |
  +----------------------------------------------------------+

  No single layer is sufficient. Each layer catches what the others miss.
```

### KMS (Key Management Service)

```
KMS KEY HIERARCHY:

  +--Customer Master Key (CMK)--+
  | - Created in KMS            |
  | - Never leaves KMS          |
  | - Used to encrypt data keys |
  | - Rotation: automatic       |
  |   (annual) or manual        |
  +-------------+---------------+
                |
  +-------------v---------------+
  | Data Encryption Key (DEK)   |
  | - Generated by KMS          |
  | - Used to encrypt your data |
  | - Plaintext used, then      |
  |   discarded from memory     |
  | - Encrypted copy stored     |
  |   with your data            |
  +-----------------------------+

ENVELOPE ENCRYPTION:
  1. KMS generates a Data Key
  2. You encrypt your data with the Data Key (plaintext)
  3. KMS encrypts the Data Key with the CMK
  4. You store the encrypted Data Key alongside your encrypted data
  5. To decrypt: KMS decrypts the Data Key, you use it to decrypt your data

KEY TYPES:
  - AWS Managed Keys:    aws/s3, aws/ebs (AWS manages, free)
  - Customer Managed Keys: You create and manage (charged per key)
  - Custom Key Store:    Keys stored in CloudHSM (hardware security module)
```

### AWS Inspector, Macie, and WAF

```
INSPECTOR: Vulnerability scanning
  - Scans EC2 instances and ECR images
  - Checks for CVEs, network exposure, software vulnerabilities
  - Continuous scanning (not just one-time)
  - Prioritizes findings by severity

MACIE: Sensitive data discovery
  - Scans S3 buckets for PII, PHI, financial data
  - Uses ML to classify data
  - Alerts on exposed sensitive data
  - Helps with GDPR, HIPAA compliance

WAF (Web Application Firewall):
  +--- Protects ALB, API Gateway, CloudFront
  |
  +--- RULES:
  |    - SQL injection protection
  |    - Cross-site scripting (XSS)
  |    - IP reputation lists
  |    - Rate limiting (DDoS layer 7)
  |    - Geo-blocking
  |    - Custom rules (regex patterns)
  |
  +--- AWS Managed Rule Groups (pre-built)
  |    - AWSManagedRulesCommonRuleSet
  |    - AWSManagedRulesSQLiRuleSet
  |    - AWSManagedRulesKnownBadInputsRuleSet

SHIELD:
  - Standard: Free DDoS protection (layer 3/4) for all AWS customers
  - Advanced: $3,000/month, layer 7 protection, DDoS response team,
    cost protection (credits for scaling charges during attack)
```

## Step-by-Step Practical

### Encrypt EBS Volumes

```bash
# Step 1: Enable default EBS encryption for the account
aws ec2 enable-ebs-encryption-by-default

# Verify
aws ec2 get-ebs-encryption-by-default
# Expected: { "EbsEncryptionByDefault": true }

# Step 2: Create a custom KMS key for EBS
KMS_KEY_ID=$(aws kms create-key \
    --description "EBS encryption key for production" \
    --key-usage ENCRYPT_DECRYPT \
    --origin AWS_KMS \
    --query 'KeyMetadata.KeyId' --output text)

aws kms create-alias \
    --alias-name alias/ebs-production \
    --target-key-id $KMS_KEY_ID

# Step 3: Create an encrypted EBS volume
VOLUME_ID=$(aws ec2 create-volume \
    --availability-zone us-east-1a \
    --size 50 \
    --volume-type gp3 \
    --encrypted \
    --kms-key-id $KMS_KEY_ID \
    --tag-specifications 'ResourceType=volume,Tags=[{Key=Name,Value=encrypted-data}]' \
    --query 'VolumeId' --output text)

echo "Encrypted volume: $VOLUME_ID"

# Step 4: Encrypt an existing unencrypted volume (via snapshot)
# Create snapshot of unencrypted volume
SNAP_ID=$(aws ec2 create-snapshot \
    --volume-id vol-UNENCRYPTED \
    --description "Snapshot for encryption" \
    --query 'SnapshotId' --output text)

aws ec2 wait snapshot-completed --snapshot-ids $SNAP_ID

# Copy snapshot with encryption
ENC_SNAP_ID=$(aws ec2 copy-snapshot \
    --source-region us-east-1 \
    --source-snapshot-id $SNAP_ID \
    --encrypted \
    --kms-key-id $KMS_KEY_ID \
    --description "Encrypted copy" \
    --query 'SnapshotId' --output text)

# Create encrypted volume from encrypted snapshot
aws ec2 create-volume \
    --availability-zone us-east-1a \
    --snapshot-id $ENC_SNAP_ID \
    --volume-type gp3

# Step 5: Set up key rotation
aws kms enable-key-rotation --key-id $KMS_KEY_ID

# Verify rotation is enabled
aws kms get-key-rotation-status --key-id $KMS_KEY_ID
# Expected: { "KeyRotationEnabled": true }
```

### Manage Secrets with Secrets Manager

```bash
# Step 1: Create a database secret
aws secretsmanager create-secret \
    --name production/database/postgres \
    --description "Production PostgreSQL credentials" \
    --secret-string '{
        "host": "production-postgres.xxxx.us-east-1.rds.amazonaws.com",
        "port": 5432,
        "username": "dbadmin",
        "password": "SuperSecretP@ssw0rd!",
        "dbname": "myapp",
        "engine": "postgres"
    }'

# Step 2: Create an API key secret
aws secretsmanager create-secret \
    --name production/api/stripe \
    --description "Stripe API key" \
    --secret-string '{"api_key": "sk_live_xxxxxxxxxxxx"}'

# Step 3: Retrieve a secret
aws secretsmanager get-secret-value \
    --secret-id production/database/postgres \
    --query 'SecretString' --output text | python3 -m json.tool
# Expected:
# {
#     "host": "production-postgres.xxxx.us-east-1.rds.amazonaws.com",
#     "port": 5432,
#     "username": "dbadmin",
#     "password": "SuperSecretP@ssw0rd!",
#     ...
# }

# Step 4: Use secrets in application code (Python example)
cat > get_secret.py << 'PYEOF'
import boto3
import json

def get_db_credentials():
    client = boto3.client('secretsmanager', region_name='us-east-1')
    response = client.get_secret_value(SecretId='production/database/postgres')
    secret = json.loads(response['SecretString'])
    return secret

# Usage
creds = get_db_credentials()
# connection_string = f"postgresql://{creds['username']}:{creds['password']}@{creds['host']}:{creds['port']}/{creds['dbname']}"
PYEOF

# Step 5: Set up automatic rotation for RDS secrets
# Create the rotation Lambda (AWS provides templates)
aws secretsmanager rotate-secret \
    --secret-id production/database/postgres \
    --rotation-lambda-arn arn:aws:lambda:us-east-1:$ACCOUNT_ID:function:SecretsManagerRotation \
    --rotation-rules '{"AutomaticallyAfterDays": 30}'

# Step 6: Use secrets in ECS task definitions (no code changes needed)
# In your task definition, reference the secret:
# "secrets": [
#     {
#         "name": "DB_PASSWORD",
#         "valueFrom": "arn:aws:secretsmanager:us-east-1:123456789012:secret:production/database/postgres:password::"
#     }
# ]
```

### Enable GuardDuty

```bash
# Enable GuardDuty
DETECTOR_ID=$(aws guardduty create-detector \
    --enable \
    --finding-publishing-frequency FIFTEEN_MINUTES \
    --query 'DetectorId' --output text)

echo "GuardDuty Detector: $DETECTOR_ID"

# List findings (may be empty initially)
aws guardduty list-findings \
    --detector-id $DETECTOR_ID \
    --finding-criteria '{
        "Criterion": {
            "severity": {
                "Gte": 7
            }
        }
    }'

# Generate sample findings for testing
aws guardduty create-sample-findings \
    --detector-id $DETECTOR_ID \
    --finding-types "Recon:EC2/PortProbeUnprotectedPort" \
                    "UnauthorizedAccess:EC2/SSHBruteForce" \
                    "CryptoCurrency:EC2/BitcoinTool.B!DNS"

# View sample findings
FINDING_IDS=$(aws guardduty list-findings \
    --detector-id $DETECTOR_ID \
    --query 'FindingIds[0:3]' --output text)

aws guardduty get-findings \
    --detector-id $DETECTOR_ID \
    --finding-ids $FINDING_IDS \
    --query 'Findings[*].[Severity,Type,Title]' \
    --output table
```

### Set Up WAF on ALB

```bash
# Step 1: Create a WAF Web ACL
WAF_ACL_ARN=$(aws wafv2 create-web-acl \
    --name production-web-acl \
    --scope REGIONAL \
    --default-action '{"Allow": {}}' \
    --visibility-config '{
        "SampledRequestsEnabled": true,
        "CloudWatchMetricsEnabled": true,
        "MetricName": "production-web-acl"
    }' \
    --rules '[
        {
            "Name": "AWSManagedRulesCommonRuleSet",
            "Priority": 1,
            "OverrideAction": {"None": {}},
            "VisibilityConfig": {
                "SampledRequestsEnabled": true,
                "CloudWatchMetricsEnabled": true,
                "MetricName": "CommonRuleSet"
            },
            "Statement": {
                "ManagedRuleGroupStatement": {
                    "VendorName": "AWS",
                    "Name": "AWSManagedRulesCommonRuleSet"
                }
            }
        },
        {
            "Name": "AWSManagedRulesSQLiRuleSet",
            "Priority": 2,
            "OverrideAction": {"None": {}},
            "VisibilityConfig": {
                "SampledRequestsEnabled": true,
                "CloudWatchMetricsEnabled": true,
                "MetricName": "SQLiRuleSet"
            },
            "Statement": {
                "ManagedRuleGroupStatement": {
                    "VendorName": "AWS",
                    "Name": "AWSManagedRulesSQLiRuleSet"
                }
            }
        },
        {
            "Name": "RateLimitRule",
            "Priority": 3,
            "Action": {"Block": {}},
            "VisibilityConfig": {
                "SampledRequestsEnabled": true,
                "CloudWatchMetricsEnabled": true,
                "MetricName": "RateLimit"
            },
            "Statement": {
                "RateBasedStatement": {
                    "Limit": 2000,
                    "AggregateKeyType": "IP"
                }
            }
        }
    ]' \
    --query 'Summary.ARN' --output text)

# Step 2: Associate WAF with ALB
aws wafv2 associate-web-acl \
    --web-acl-arn $WAF_ACL_ARN \
    --resource-arn $ALB_ARN
```

## Knowledge Check

### Question 1
What is envelope encryption and why does KMS use it?

**Answer:** Envelope encryption is a strategy where data is encrypted with a Data
Encryption Key (DEK), and the DEK itself is encrypted with a master key (CMK) stored
in KMS. KMS uses this because: (1) encrypting large data directly with the CMK would
require sending all data to KMS over the network, which is slow and expensive, (2)
with envelope encryption, only the small DEK is sent to KMS for encryption/decryption,
(3) the CMK never leaves KMS hardware, providing better security, (4) you can use
different DEKs for different data while managing a single CMK.

### Question 2
What is the difference between Secrets Manager and Systems Manager Parameter Store?

**Answer:** Secrets Manager is purpose-built for secrets: it supports automatic
rotation (via Lambda), native RDS/Redshift/DocumentDB integration, costs $0.40 per
secret per month + $0.05 per 10,000 API calls. Parameter Store can store secrets
(SecureString type encrypted with KMS) but does not support automatic rotation. It is
free for standard parameters (up to 10,000) and supports hierarchical storage
(/app/prod/db/password). Use Secrets Manager for database credentials and API keys
that need rotation. Use Parameter Store for configuration values and secrets where
rotation is not needed.

### Question 3
How does GuardDuty detect threats without installing agents?

**Answer:** GuardDuty analyzes data from existing AWS data sources: CloudTrail
management and data events (API calls), VPC Flow Logs (network traffic metadata),
DNS query logs (domain lookups), S3 data events, EKS audit logs, and RDS login
activity. It uses machine learning, anomaly detection, and integrated threat
intelligence (known malicious IPs, domains) to identify suspicious activity. No
agents, no infrastructure, no performance impact on your workloads.

### Question 4
When would you use AWS Shield Advanced versus just WAF?

**Answer:** AWS Shield Standard (free) provides basic DDoS protection at layers 3
and 4 for all AWS customers. WAF protects against layer 7 attacks (SQL injection,
XSS, rate limiting). Shield Advanced ($3,000/month) is for organizations that need:
(1) protection against larger, more sophisticated DDoS attacks, (2) 24/7 access to
the AWS DDoS Response Team (DRT), (3) DDoS cost protection (AWS credits you for
scaling charges caused by DDoS attacks), (4) real-time visibility into attacks, (5)
advanced attack diagnostics. Most organizations need WAF; Shield Advanced is for
high-value targets.

### Question 5
How should you handle database passwords in a CI/CD pipeline?

**Answer:** Never store passwords in source code, environment variables in the
pipeline definition, or configuration files. Instead: (1) Store the password in
Secrets Manager, (2) grant the CI/CD pipeline's IAM role permission to read the
specific secret, (3) have the pipeline retrieve the secret at runtime using the
Secrets Manager API, (4) pass the secret to the application via ECS task definition
secrets or Lambda environment variables encrypted with KMS, (5) enable automatic
rotation so the password changes regularly without any code changes.
